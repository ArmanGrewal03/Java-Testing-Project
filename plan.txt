2. PROJECT REQUIREMENTS THAT DRIVE THIS PLAN
=====================================================================
The official project description imposes the following major requirements that directly affect implementation:

2.1 Core Project Requirements
- The selected project must be an open-source Java application/library of meaningful size.
- The group must test the selected project using course-based testing methods.
- The final report must contain the finalized test plan, findings, results, critical analysis, barriers, and tool-based evidence.

2.2 Section 5 Requirements
According to the project description, the group must apply each required testing abstraction/approach to selected methods and implement actual testing programs. The course requires using 4 different testing abstractions out of the available 5. Since the group test plan already commits to the following four, this project implementation will use:
- Input Space Partitioning (ISP) with Boundary Value Analysis (BVA)
- Graph-Based Testing (CFG and DFG)
- Logic-Based Testing
- Mutation Testing

Formal methods are not included in the group plan and therefore are out of scope for this implementation.

2.3 LLM-Assisted Testing Requirement
In addition to regular testing, the group must complete an LLM-assisted testing experiment using ChatUniTest and a supported local LLM. This must include:
- selecting 3 to 5 Java methods,
- manually designing tests first,
- generating LLM-assisted tests through ChatUniTest,
- comparing manual vs LLM-generated tests using structural coverage and mutation evaluation,
- documenting the process, commands, and limitations,
- and including the full LLM interaction history in the appendix as required.

2.4 Submission Requirements
The final project deliverables must include:
- Final written report as one PDF
- ZIP file containing project code, tests, graphs, screenshots, coverage, mutation evidence, and supporting artifacts
- Report examples should be selective, but the implementation work must still be done comprehensively
- The final report must include critical reflection, barriers, test justification, tool justification, and LLM-assisted testing analysis

3. SYSTEM UNDER TEST (SUT)
=====================================================================
Project: Joda-Time
Version: 2.14.0
Language: Java
Build Tool: Maven
Repository: https://github.com/JodaOrg/joda-time

Why Joda-Time is suitable:
- It is large enough for the project scope.
- It contains approximately 166 classes according to the test plan.
- It has well-defined, widely used date/time methods with meaningful edge cases.
- It includes arithmetic behavior, boundary-sensitive logic, exception paths, immutability behavior, and interval logic.
- It is testable at the unit level with JUnit and Maven.

Important testing characteristics of Joda-Time that must shape implementation:
- Many classes are immutable.
- Date/time logic is highly sensitive to boundaries.
- Leap years, month-end rollover, and interval boundary handling are common fault sources.
- Arithmetic operations can overflow or behave unexpectedly on special values.
- Null handling and exception behavior are important in some methods.
- Time-zone dependence must be controlled to ensure deterministic tests.

=====================================================================
5. OVERALL IMPLEMENTATION STRATEGY
=====================================================================
The project should be implemented in layers, not all at once. The order matters.

PHASE ORDER:
1. Environment setup and repository preparation
2. Source code reading and method-under-test analysis
3. Manual test design for all 10 methods
4. JUnit implementation of manual tests
5. CFG / DFG / logic artifact creation
6. Coverage execution and refinement
7. Mutation testing execution and refinement
8. LLM-assisted testing experiment
9. Comparative analysis and final evidence collection
10. Final report assembly and ZIP packaging

This ordering is important because:
- coverage and mutation testing only make sense after stable manual tests exist,
- LLM-generated tests must be compared against a manual baseline,
- diagrams and logic analysis should reflect the actual implemented tests,
- report writing should happen after evidence is collected, not before.

6.3 TASK GROUP C – INPUT SPACE PARTITIONING (ISP) + BVA DESIGN
---------------------------------------------------------------------
Objective:
Define high-quality manual tests using input space partitioning and boundary value analysis.

This task applies to all 10 methods.

For EACH method:
1. Identify input parameters and relevant object state.
2. Partition inputs into equivalence classes.
3. Mark valid partitions and invalid/error partitions.
4. Identify boundaries for each partition.
5. Select representative values.
6. Convert the chosen values into concrete JUnit-ready test cases.
7. Note expected outputs or exceptions.

Method-specific design reminders:

TI-01 DateTime.withYear(int)
Potential partitions:
- ordinary valid year
- leap year preserving valid date
- year causing invalid date transition
- extreme values
- negative years / year zero if meaningful
Check:
- returned year updated correctly
- original object unchanged
- exception behavior if applicable

TI-02 DateTime.plusDays(int)
Potential partitions:
- positive days
- zero days
- negative days
- month boundary crossing
- year boundary crossing
- leap year crossing

TI-03 LocalDate.plusMonths(int)
Potential partitions:
- positive months
- zero months
- negative months
- month-end clamping cases
- leap-year cases
- year rollover cases

TI-04 LocalDate.minusDays(int)
Potential partitions:
- positive subtraction
- zero subtraction
- negative subtraction (effectively add)
- month/year boundary crossings

TI-05 Duration.plus(long)
Potential partitions:
- positive addition
- zero addition
- negative addition
- large addition
- overflow-sensitive values

TI-06 Duration.dividedBy(long)
Potential partitions:
- divisor = 1
- divisor = -1
- divisor = 0 (exception)
- positive divisor > 1
- negative divisor < -1
- large magnitude divisor

TI-07 Period.plus(ReadablePeriod)
Potential partitions:
- normal non-null period
- null period if supported/ignored
- zero period
- mixed fields
- large field combinations

TI-08 Period.normalizedStandard()
Potential partitions:
- already normalized period
- non-normalized period
- zero-valued period
- large value period
- cases where normalization changes field structure

TI-09 Interval.contains(long)
Potential partitions:
- instant before start
- instant equal to start
- instant inside interval
- instant equal to end
- instant after end
- zero-length interval case

TI-10 Interval.overlaps(ReadableInterval)
Potential partitions:
- overlapping intervals
- touching but not overlapping intervals
- disjoint intervals
- one interval inside another
- equal intervals
- null interval behavior if relevant

Deliverables from this task:
- partition tables for all 10 methods,
- boundary tables for all 10 methods,
- concrete test case list for all 10 methods.

6.6 TASK GROUP F – LOGIC-BASED TESTING
---------------------------------------------------------------------
Objective:
Analyze predicate-heavy methods and design tests that exercise logical clauses meaningfully.

Best candidates:
- TI-09 Interval.contains(long)
- TI-10 Interval.overlaps(ReadableInterval)
- TI-06 Duration.dividedBy(long)

Task steps:
1. Extract predicates from method source.
2. Rewrite compound conditions into explicit Boolean form.
3. Identify clauses.
4. Determine desired logic coverage criterion for practical analysis.
5. Select test values so each clause influences outcome where meaningful.
6. Confirm tests exercise true/false outcomes and important boundary combinations.

Examples:
TI-09 Interval.contains(long)
Predicate idea:
(millisInstant >= start) AND (millisInstant < end)
Need tests that isolate:
- start-boundary inclusion,
- end-boundary exclusion,
- before start,
- strictly inside,
- after end.

TI-10 Interval.overlaps(ReadableInterval)
Need tests for:
- true overlap,
- touching only,
- fully separate,
- one interval contained within another,
- equal intervals if allowed.

TI-06 Duration.dividedBy(long)
Need tests for:
- divisor equals one branch,
- divisor zero exception branch,
- ordinary division branch.

Deliverables from this task:
- predicate and clause breakdowns,
- logic-based test requirement tables,
- matching implemented test cases,
- report-ready logic examples.

Evidence to save:
- predicate tables,
- clause truth assignments,

6.8 TASK GROUP H – MUTATION TESTING
---------------------------------------------------------------------
Objective:
Evaluate the strength of the manual test suite using mutation testing and strengthen tests where mutants survive.

Primary method:
- Use PIT if environment supports it.

Fallback method:
- manual mutant experiments if PIT fails or is incompatible.

PIT task steps:
1. Add/configure pitest-maven plugin.
2. Set target classes and target tests.
3. Run mutation coverage on the relevant packages/classes.
4. Review killed vs survived mutants.
5. Identify weaknesses in the current tests.
6. Add tests that specifically target surviving mutant behavior.
7. Re-run PIT and compare improvement.

Mutation opportunities by method:
- relational changes in interval boundaries
- arithmetic replacement in duration or period arithmetic
- constant replacements
- exception-check removal in division or invalid-state logic
- off-by-one boundary changes

If using manual mutants, create specific controlled mutants such as:
- change >= to > in contains logic,
- remove divisor zero guard in dividedBy,
- alter arithmetic constant or return expression,
- alter month handling logic assumptions.

For each manual mutant, record:
- mutant ID,
- original code,
- mutated code,
- expected semantic difference,
- which tests kill it,
- whether killed by manual tests only or also by LLM tests later.

Deliverables from this task:
- PIT report or manual mutant experiment table,
- mutation score summary,
- analysis of surviving mutants,
- strengthened tests where appropriate.

Evidence to save:
- PIT HTML report,
- screenshots of killed/survived mutants,
- manual mutant experiment logs if fallback used.

13. RISKS DURING IMPLEMENTATION AND CONTINGENCY PLAN
=====================================================================
Risk 1: Maven or dependency setup issues
Mitigation:
- finalize environment early,
- test build before writing many tests,
- keep a working branch.

Risk 2: JaCoCo or PIT compatibility issues
Mitigation:
- configure tools early,
- run on a small target first,
- use manual mutant fallback if PIT fails.

Risk 3: Low branch/mutation effectiveness
Mitigation:
- review uncovered branches,
- add boundary and exception tests,
- use mutation survivors to guide new tests.

Risk 4: LLM-generated tests fail to compile or are poor quality
Mitigation:
- document failures as part of qualitative analysis,
- fix only minor issues needed to run,
- use this as evidence of limitations rather than hiding it.

Risk 5: Team members drift or duplicate work
Mitigation:
- maintain a central checklist,
- assign method ownership clearly,
- review progress every lab/week.

Risk 6: Report becomes too descriptive and not evidence-based
Mitigation:
- collect screenshots and artifacts as testing proceeds,
- keep a results folder organized from the beginning.
